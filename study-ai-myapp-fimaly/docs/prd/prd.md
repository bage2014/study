# 家庭族谱APP产品需求文档

## 1. 产品概述

### 1.1 产品简介

家庭族谱APP是一款用于记录、管理和展示家族成员信息的应用，支持Web端和Android端。通过该应用，用户可以创建完整的家族树，记录家族历史和重要事件，上传和管理家族多媒体资料，实现家族文化的传承和分享。

### 1.2 产品目标

- 帮助用户便捷地记录和管理家族成员信息
- 提供直观的家族树可视化展示
- 支持家族历史和重要事件的记录
- 实现家族多媒体资料的管理和分享
- 保障用户数据安全和隐私

### 1.3 目标用户

- 普通家庭用户：希望记录和管理家族成员信息，了解家族历史
- 家谱研究者：需要专业的工具来整理和分析家族数据
- 老年人：希望为后代留下家族历史记录
- 年轻人：对家族历史感兴趣，希望通过现代化工具了解家族背景

### 1.4 竞品分析

6. **本地数据备份**：支持离线使用和本地数据备份，确保数据安全
7. **家族文化传承**：提供家族文化、传统习俗的记录和传承功能

## 3. 核心流程

### 3.1 用户注册流程

1. 用户打开应用，点击注册按钮
2. 输入手机号/邮箱、密码、验证码
3. 点击注册，系统验证信息
4. 注册成功，自动登录进入首页

### 3.2 家族管理流程

#### 3.2.1 新建家族流程
1. 用户登录后，点击创建家族
2. 输入家族名称、家族描述
3. 上传家族头像（可选）
4. 点击创建，系统生成家族空间
5. 进入家族管理页，开始添加成员

#### 3.2.2 退出家族流程
1. 进入家族管理页
2. 点击退出家族按钮
3. 系统提示确认
4. 确认后，用户退出家族
5. 返回到首页

#### 3.2.3 编辑家族流程
1. 进入成员管理页，选择家族
2. 点击查看/编辑家族按钮
3. 在弹出的模态框中修改家族信息
4. 点击保存，家族信息更新成功

#### 3.2.4 查看家族流程
1. 进入成员管理页，选择家族
2. 查看家族信息卡片
3. 显示家族名称、创建时间、描述等信息

### 3.3 成员管理流程

#### 3.3.1 查看家族成员流程
1. 进入成员管理页，选择家族
2. 系统显示该家族的成员列表
3. 支持按成员信息进行排序和筛选

#### 3.3.2 新建成员流程
1. 进入成员管理页，选择家族
2. 点击添加成员按钮
3. 系统验证是否已选择家族，若未选择则提示用户
4. 输入成员基本信息（姓名、性别、出生日期、去世日期、手机号、邮箱、详细信息等）
5. 上传成员照片（可选）
6. 点击保存，成员添加成功
7. 成员列表自动更新
8. 家族树自动更新

#### 3.3.3 备注成员流程
1. 进入成员详情页
2. 编辑成员的详细信息字段
3. 添加备注信息
4. 点击保存，备注信息更新成功

#### 3.3.4 绑定成员关系流程
1. 进入关系管理页
2. 选择成员1和成员2
3. 支持通过搜索页面选择成员
4. 选择关系类型
5. 点击保存，关系绑定成功
6. 家族树自动更新

### 3.4 家族树页面流程

1. 进入家族树页
2. 查看家族列表，选择一个家族
3. 查看当前家族的所有成员和关联关系
4. 点击右上角"新建家族"按钮，创建新家族
5. 点击右上角"查看明细"按钮，查看和编辑当前家族
6. 在当前家族下，点击"添加成员"按钮，添加新成员
7. 在当前家族下，点击"添加关系"按钮，维护成员关联关系
8. 支持删除成员和删除关系
9. 查看家族成员列表和关系列表

### 3.5 成员搜索流程

1. 进入成员搜索页
2. 输入手机号或邮箱进行搜索
3. 查看搜索结果列表
4. 点击"选择"按钮选择成员
5. 系统将选择的成员信息带回原页面

### 3.6 关系管理流程

1. 进入关系管理页
2. 选择成员1和成员2
3. 支持通过搜索页面选择成员
4. 选择关系类型
5. 点击保存，关系添加成功
6. 家族树自动更新

## 5. 技术实现

### 5.1 技术栈

- **后端**：Spring Boot 3.1+
- **数据库**：H2 in-memory (开发环境)，MySQL 8.0+ (生产环境)
- **Web前端**：Vue 3+、Pinia、Tailwind CSS
- **移动端**：Flutter
- **云存储**：对象存储服务（如OSS）
- **认证**：JWT

### 5.2 架构设计

- **后端**：RESTful API架构，分层设计（Controller、Service、Repository）
- **前端**：单页应用（SPA），组件化设计
- **数据同步**：前后端分离，通过API交互
- **数据存储**：结构化数据存储在MySQL，多媒体文件存储在对象存储服务

### 5.3 API设计规范

- **参数传递**：
  - GET请求：使用查询参数（query parameters）传递参数
  - POST/PUT请求：使用请求体（request body）传递参数
  - **禁止**：将参数直接放在URL路径中，例如 `/families/{familyId}/members` 这种形式是不允许的
  - **正确示例**：
    - GET请求：`/members/family?familyId=1`
    - POST请求：`/members` 配合请求体 `{ "familyId": 1, "name": "张三" }`

### 5.4 安全设计

- **用户认证**：JWT token认证
- **权限管理**：基于角色的访问控制（RBAC）
- **路由拦截**：未登录用户或权限不足（403错误）时自动跳转到登录页面
- **数据加密**：敏感数据加密存储
- **传输安全**：HTTPS协议
- **防止SQL注入**：使用参数化查询
- **防止XSS攻击**：输入验证和输出编码

## 6. 数据需求

### 6.1 核心数据模型

1. **用户表**：用户ID、手机号/邮箱、密码、昵称、头像
2. **家族表**：家族ID、家族名称、描述、创建时间、创建者ID
3. **成员表**：成员ID、家族ID、姓名、性别、出生日期、去世日期、照片、详细信息、手机号、邮箱
4. **关系表**：关系ID、成员ID1、成员ID2、关系类型
5. **历史事件表**：事件ID、家族ID、事件名称、描述、发生时间、相关成员、照片
6. **多媒体表**：媒体ID、家族ID、类型、URL、描述、上传时间、上传者ID
7. **权限表**：权限ID、用户ID、家族ID、权限级别

### 6.2 数据存储

- **结构化数据**：MySQL数据库
- **多媒体数据**：对象存储服务
- **备份策略**：定期数据备份，支持手动备份和自动备份

## 7. 非功能需求

### 7.1 性能需求

- **响应时间**：页面加载时间<3秒，API响应时间<500ms

### 7.2 可用性需求

- **系统可用性**：99.9%
- **数据可用性**：数据多重备份，确保数据不丢失
- **故障恢复**：系统故障后30分钟内恢复服务

### 7.3 安全性需求

- **数据安全**：敏感数据加密存储，传输加密
- **用户隐私**：严格保护用户隐私，未经授权不得访问
- **权限控制**：精细化的权限管理，确保数据安全

### 7.4 可扩展性需求

- **功能扩展**：支持模块化开发，便于功能扩展
- **性能扩展**：支持水平扩展，应对用户量增长
- **技术升级**：支持技术栈的平滑升级

## 10. 风险评估

### 10.1 潜在风险

- **技术风险**：技术栈选择不当，可能导致开发效率低下或性能问题
- **数据风险**：数据量过大，可能导致系统性能下降或存储成本增加
- **用户风险**：用户接受度低，可能导致产品推广困难
- **安全风险**：数据安全措施不当，可能导致用户数据泄露

### 10.2 风险应对策略

- **技术风险**：选择成熟稳定的技术栈，进行充分的技术评估和测试
- **数据风险**：优化数据库设计，采用分库分表策略，合理规划存储方案
- **用户风险**：进行用户调研，了解用户需求，提供良好的用户体验
- **安全风险**：加强安全措施，定期进行安全审计和漏洞扫描
