## 1. 技术选型调研

### 1.1 工作流引擎选型

#### 1.1.1 选型对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **Temporal** | 持久化工作流、Signal 机制、断点续跑、0 资源等待、社区活跃 | 需要部署 Temporal Server、学习曲线较陡 | 企业级可靠性要求、需要人工审批、长流程编排 |
| **Airflow** | 成熟生态、丰富的操作符、可视化 DAG | 批处理导向、非持久化、复杂流程表达困难 | 数据管道、定时任务、批处理场景 |
| **AWS Step Functions** | 完全托管、与 AWS 深度集成、可视化 | 绑定 AWS 生态、成本较高、自定义能力有限 | AWS 云环境、简单工作流、快速上线 |
| **Camunda** | BPMN 标准、可视化流程设计、成熟稳定 | 重量级、学习曲线陡、部署复杂 | 企业级 BPM、合规流程、审计要求高 |
| **Spring StateMachine** | 轻量级、Spring 生态集成、易于上手 | 非持久化、功能有限、社区较小 | 简单状态机场景、轻量级流程 |

#### 1.1.2 Temporal 选型理由

**核心优势**：

1. **持久化工作流**：Temporal 自动持久化工作流状态，支持宕机恢复和断点续跑
2. **Signal 机制**：支持外部信号触发（如人工审批），等待期间 Worker 线程释放，0 资源消耗
3. **Activity 模式**：将业务逻辑封装为 Activity，支持重试、超时、错误处理
4. **分布式执行**：支持多 Worker 分布式部署，自动负载均衡
5. **版本控制**：支持工作流版本升级，旧实例继续使用旧代码

**技术指标对比**：

| 指标 | Temporal | Airflow | Step Functions |
|------|----------|---------|---------------|
| 状态持久化 | ✅ 自动持久化 | ❌ 非持久化 | ✅ 自动持久化 |
| 外部信号 | ✅ Signal | ❌ 不支持 | ✅ Task Token |
| 断点续跑 | ✅ 支持 | ❌ 不支持 | ✅ 支持 |
| 等待资源消耗 | 0 | 持续占用 | 低 |
| 分布式部署 | ✅ 支持 | ✅ 支持 | ✅ 托管 |
| 社区活跃度 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |

**结论**：Temporal 是唯一满足"持久化工作流 + Signal 审批 + 0 资源等待"三项核心需求的方案。

### 1.2 AI 框架选型

#### 1.2.1 选型对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **LangChain4j** | Java 原生、轻量级、Agent 模式、工具调用、多模型支持 | 生态不如 Python 版丰富、文档较少 | Java 后端集成、轻量级 Agent、工具调用 |
| **LangChain (Python)** | 生态丰富、文档完善、社区活跃、大量第三方集成 | 需要 Python 环境、与 Java 后端集成复杂 | Python 生态、复杂 Agent、快速原型 |
| **Semantic Kernel** | Microsoft 出品、多语言支持、插件系统 | 文档有限、社区较小、功能不如 LangChain | .NET 生态、Microsoft 技术栈 |
| **Haystack** | 专注 RAG、文档处理、管道式设计 | Agent 能力有限、学习曲线较陡 | RAG 场景、文档问答、知识检索 |

#### 1.2.2 LangChain4j 选型理由

**核心优势**：

1. **Java 原生**：与 Spring Boot 无缝集成，无需额外语言环境
2. **Agent 模式**：支持 ReAct 模式，自动选择和调用工具
3. **工具调用**：内置工具调用框架，易于扩展自定义工具
4. **多模型支持**：支持 Claude、OpenAI、Ollama 等多种 LLM
5. **轻量级**：核心库小巧，启动快速，适合生产环境

**技术指标对比**：

| 指标 | LangChain4j | LangChain (Python) | Semantic Kernel |
|------|-------------|-------------------|-----------------|
| Java 集成 | ✅ 原生 | ❌ 需要 RPC | ✅ 支持 |
| Agent 模式 | ✅ ReAct | ✅ 多种模式 | ✅ Planner |
| 工具调用 | ✅ 内置 | ✅ 丰富 | ✅ 插件 |
| 多模型支持 | ✅ 支持 | ✅ 丰富 | ✅ 支持 |
| 学习曲线 | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |

**结论**：LangChain4j 是 Java 生态中最成熟的 AI Agent 框架，满足项目的工具调用和多模型切换需求。

## 2. 设计原理分析

### 2.1 三层分解设计原理

#### 2.1.1 设计背景

传统的 AI 代码生成面临以下挑战：

1. **上下文过长**：完整项目代码 + 需求描述可能超出 LLM 上下文限制
2. **指令模糊**：一次性生成时，AI 需要自行拆解需求并分配代码到多个文件
3. **冲突概率高**：多模块同时修改容易产生代码冲突
4. **失败成本高**：任何部分失败都需要全部重生成

#### 2.1.2 三层分解策略

```
需求描述（自然语言）
    │
    ▼
第一层：需求分析 → 功能点拆分（3~8 个）
    │
    ▼
第二层：功能点 → 原子任务拆分（每个功能点 1~3 个）
    │
    ▼
第三层：原子任务 → 代码生成（每次只处理 1~3 个目标文件）
```

#### 2.1.3 设计优势

| 维度 | 一次性生成 | 三层分解 | 优势 |
|------|-----------|---------|------|
| LLM 上下文 | 全量需求 + 整个项目结构 | 每个任务只处理 1~3 个文件 | 降低上下文长度 80%+ |
| 指令聚焦度 | 低（AI 自行拆解） | 高（精确目标文件） | 提高生成精度 |
| 失败恢复 | 全部重生成 | 仅失败任务重执行 | 降低失败成本 |
| 代码冲突 | 高（多模块同时修改） | 低（独立 diff） | 减少冲突 |
| 复杂需求成功率 | 低（上下文超限） | 高（短上下文） | 提高成功率 |

#### 2.1.4 实现细节

**功能点拆分规则**：
- 每个功能点对应一个独立的业务模块或功能域
- 功能点数量控制在 3~8 个，避免过细或过粗
- 每个功能点包含验收标准（Acceptance Criteria）

**原子任务拆分规则**：
- 每个原子任务对应一个或多个具体文件的修改
- 任务数量控制在 1~3 个/功能点
- 任务类型分为：CONTROLLER、SERVICE、REPOSITORY、ENTITY、CONFIG 等

**代码生成模式**：

| 模式 | 触发条件 | 精度 |
|------|---------|------|
| 原子任务聚焦 | currentTask != null | 最高，只处理指定文件 |
| 测试失败修复 | testFailureContext != null | 高，最小化变更 |
| 全量需求生成 | 其他情况 | 兜底模式 |

### 2.2 AgentToolRegistry 设计原理

#### 2.2.1 设计背景

AI Agent 需要调用多种工具完成任务，工具的注册和管理面临以下挑战：

1. **工具分散**：工具分布在不同类中，难以统一管理
2. **注册繁琐**：每次新增工具需要修改 Service 代码
3. **类型安全**：工具调用需要类型转换，容易出错

#### 2.2.2 设计方案

**核心设计**：

```
AgentTool（标记接口）
    └── toolName(): String

@Component FileReadTool   → implements AgentTool, toolName="file-read"
@Component FileWriteTool  → implements AgentTool, toolName="file-write"
@Component GitTool        → implements AgentTool, toolName="git"
@Component MavenTool      → implements AgentTool, toolName="maven"
@Component NpmTool        → implements AgentTool, toolName="npm"
@Component DockerTool     → implements AgentTool, toolName="docker"
@Component AstAnalysisTool → implements AgentTool, toolName="ast-analysis"

AgentToolRegistry（Spring @Component）
    构造时：List<AgentTool> agentTools → Map<toolName, tool>
    getTools(String... names) → Object[]（按名称提取工具数组）
```

#### 2.2.3 设计优势

| 维度 | 传统方式 | AgentToolRegistry |
|------|---------|-------------------|
| 新增工具成本 | 修改 Service 代码 | 仅需 @Component + 实现接口 |
| 工具管理 | 分散在各 Service | 统一 Registry 管理 |
| 类型安全 | 手动类型转换 | 自动注入，类型安全 |
| 可测试性 | 难以 Mock | 易于注入 Mock 工具 |

#### 2.2.4 使用示例

```java
@Autowired
private AgentToolRegistry toolRegistry;

public void generateCode() {
    AiServices.builder(CodeGenAiService.class)
        .tools(toolRegistry.getTools("file-read", "file-write", "ast-analysis"))
        .build();
}
```

### 2.3 策略模式设计原理

#### 2.3.1 设计背景

不同企业和环境可能需要不同的构建、部署和测试策略，硬编码策略会导致：

1. **维护困难**：修改策略需要修改核心代码
2. **环境耦合**：生产环境和开发环境策略不同
3. **扩展困难**：新增策略需要修改多处代码

#### 2.3.2 设计方案

**策略接口设计模式**：

```
接口（定义契约）
    └── 默认实现（@ConditionalOnMissingBean）
    └── 可选实现（@ConditionalOnProperty）
            │
            注入到 ActivityImpl（构造器注入，不感知具体实现）
```

**四种策略维度**：

| 策略接口 | 默认实现 | 可选实现 | 激活方式 |
|---------|---------|---------|---------|
| DeployStrategy | LocalDockerDeployStrategy | - | 代码扩展 |
| BuildStrategy | LocalBuildStrategy | - | 代码扩展 |
| TestExecutionStrategy | LocalTestExecutionStrategy | - | 代码扩展 |
| PrCreationStrategy | NonePrCreationStrategy | - | 代码扩展 |

#### 2.3.3 设计优势

| 维度 | 硬编码 | 策略模式 |
|------|-------|---------|
| 修改策略 | 修改核心代码 | 修改配置或新增实现类 |
| 环境适配 | 需要分支判断 | 通过配置切换实现 |
| 新增策略 | 修改多处代码 | 仅需实现接口 |
| 代码耦合 | 高耦合 | 低耦合，符合开闭原则 |

#### 2.3.4 实现示例

```java
public interface BuildStrategy {
    BuildResult build(BuildInput input);
}

@Component
@ConditionalOnMissingBean
public class LocalBuildStrategy implements BuildStrategy {
    @Override
    public BuildResult build(BuildInput input) {
        // 本地构建逻辑
    }
}

@Component
@ConditionalOnProperty(name = "build.strategy", havingValue = "docker")
public class DockerBuildStrategy implements BuildStrategy {
    @Override
    public BuildResult build(BuildInput input) {
        // Docker 构建逻辑
    }
}
```

### 2.4 测试失败自修复设计原理

#### 2.4.1 设计背景

AI 生成的代码可能存在测试失败的情况，传统方式需要人工介入修复，影响自动化流程。

#### 2.4.2 设计方案

**自修复循环流程**：

```
TestExecutionActivity
    ├── 成功 → 继续下一阶段
    └── 失败 → testFailureContext = 错误信息
                  │
                  ├── 已达 maxTestFixRetries？→ 等待人工审批或抛出异常
                  └── 否 → CodeGenerationActivity（fixCode 模式）
                            → 输入：测试失败上下文
                            → 输出：修复后的代码
                            → WriteAndCommitActivity（fix commit）
                            → 重试 TestExecutionActivity
```

#### 2.4.3 设计优势

| 维度 | 人工修复 | 自动修复 |
|------|---------|---------|
| 修复速度 | 分钟级到小时级 | 秒级到分钟级 |
| 人工干预 | 必须人工介入 | 可选人工介入 |
| 修复质量 | 依赖工程师水平 | 依赖 AI 模型能力 |
| 流程中断 | 流程中断等待修复 | 流程自动继续 |

#### 2.4.4 关键参数

| 参数 | 默认值 | 说明 |
|------|-------|------|
| maxTestFixRetries | 3 | 最大修复重试次数 |
| autoFixOnTestFail | true | 是否启用自动修复 |

## 3. 架构设计决策

### 3.1 模块划分决策

#### 3.1.1 模块职责

| 模块 | 职责 | 依赖关系 |
|------|------|---------|
| ai-pipeline-core | 纯 Java 模块，定义接口、DTO、枚举、Workflow 编排 | 无 Spring 依赖 |
| ai-pipeline-agent | AI Agent 层，实现工具调用、模型集成 | 依赖 core |
| ai-pipeline-api | Spring Boot 主应用，实现 Activity、提供 REST API | 依赖 core、agent |
| ai-pipeline-gateway | API 网关，提供统一入口、CORS 处理 | 依赖 core（契约层） |
| ai-pipeline-ui | Vue 前端界面 | 无后端依赖 |

#### 3.1.2 划分原则

1. **契约先行**：api 模块定义接口契约，gateway 只依赖 api 模块
2. **依赖单向**：core → agent → api，避免循环依赖
3. **无 Spring 依赖的 core**：core 模块不依赖 Spring，便于测试和复用
4. **网关隔离**：gateway 作为对外入口，隔离内部服务

#### 3.1.3 决策理由

| 原则 | 说明 |
|------|------|
| **高内聚低耦合** | 每个模块职责单一，减少模块间依赖 |
| **可测试性** | core 模块无 Spring 依赖，易于单元测试 |
| **可扩展性** | 新增功能只需修改对应模块，不影响其他模块 |
| **部署灵活** | 各模块可独立部署，支持水平扩展 |

### 3.2 通信方式决策

#### 3.2.1 通信架构

```
UI (8082) → Gateway (8083) → API (8080)
              │                    │
              └── WebClient        └── Temporal SDK
                                     │
                                     └── Temporal Server (7233)
```

#### 3.2.2 通信协议

| 层级 | 协议 | 说明 |
|------|------|------|
| UI → Gateway | HTTP/REST | 标准 RESTful API |
| Gateway → API | HTTP/REST (WebClient) | 异步非阻塞调用 |
| API → Temporal | Temporal SDK | 工作流编排 |

#### 3.2.3 决策理由

| 协议 | 选择理由 |
|------|---------|
| REST | 标准化、易于调试、广泛支持 |
| WebClient | 响应式非阻塞、性能优秀、与 Spring 集成 |
| Temporal SDK | 持久化工作流、Signal 机制、断点续跑 |

### 3.3 数据存储决策

#### 3.3.1 存储方案

| 数据类型 | 存储方式 | 说明 |

#### 3.3.2 决策理由

|---------|---------|--------|----------|
| 流水线状态 | ❌ 查询性能差 | ✅ 结构化查询 | ✅ 持久化 |

## 4. 性能优化策略

### 4.1 LLM 调用优化

| 策略 | 说明 | 预期效果 |
|------|------|---------|

| **增量分析** | 只分析变更的文件，不重复分析 | 减少代码分析时间 |
| **缓存机制** | 缓存代码分析结果和生成模板 | 减少重复计算 |

## 6. 总结

### 6.1 技术选型总结

| 组件 | 选型 | 核心原因 |
|------|------|---------|
| 工作流引擎 | Temporal | 持久化工作流、Signal 审批、0 资源等待 |
| AI 框架 | LangChain4j | Java 原生、Agent 模式、工具调用 |
| 代码分析 | JavaParser | Java 原生、AST 支持、API 友好 |
| 数据库 | H2 | 零配置启动、开发友好、易于迁移 |

### 6.2 设计决策总结

| 设计点 | 决策 | 优势 |
|------|------|------|
| 三层分解 | 需求→功能点→原子任务 | 提高生成精度，降低失败成本 |
| 工具自注册 | AgentTool + Registry | 零代码扩展，统一管理 |
| 策略模式 | 可插拔策略接口 | 改配置不改代码，灵活适配 |
| 测试自修复 | 自动修复循环 | 减少人工干预，提高交付成功率 |
