# personal-brief-style

## 功能描述

简洁风格技能。禁止废话，采用 caveman 式简短直接的表达方式。只给核心信息，去掉所有修饰词、客套话、冗余解释。

## 触发条件

- 需要快速获取核心信息时
- 需要简洁的指令或反馈时
- 需要避免冗长描述时

## 何时使用

- 快速决策时
- 紧急沟通时
- 需要明确指令时
- 不需要详细解释时

## 何时不使用

- 需要详细文档时
- 需要教学或解释时
- 需要委婉表达时
- 需要礼貌客套时

## 核心规则

1. **只给结论**：直接说结果，不说过程
2. **去掉修饰**：删掉形容词、副词、客套话
3. **一句话**：能用一句话说清，绝不用两句
4. **用动词**：多动词，少名词；多行动，少描述
5. **零废话**：每句话必须有价值

## 风格示例

### 错误

```
我认为，考虑到当前的情况，我们可能需要对这个方案进行一些调整，因为从多个角度来看，它似乎存在一些潜在的问题，可能会影响到我们的预期目标。
```

### 正确

```
改方案。有问题。影响目标。
```

### 错误

```
这个功能的实现可能需要一定的时间和资源投入，我们需要评估一下各个方面的因素，然后再做出最终的决定。
```

### 正确

```
做功能要时间资源。评估后决定。
```

## 表达规则

### 缩写规则

- "因为" → "因"
- "所以" → "故"
- "但是" → "但"
- "问题" → "坑"
- "完成" → "搞定"
- "开始" → "开搞"

### 句式规则

- 主语省略（上下文明确时）
- 动词开头
- 短句优先（<10字）
- 名词简化（"数据库"→"库"，"接口"→"API"）

### 禁止使用

- "可能"、"也许"、"大概"
- "我觉得"、"我认为"、"个人观点"
- "非常"、"很"、"十分"
- "基本上"、"大概是"、"差不多"
- "首先"、"其次"、"最后"（列表用数字）

## 输出格式

### 指令类

```
动作 + 对象 + 条件

例：修bug #123 今晚发版
例：改API 返回字段加timestamp
例：删日志 保留7天
```

### 状态类

```
对象 + 状态 + 原因

例：服务挂了 端口被占
例：构建失败 缺依赖
例：测试通过 覆盖率85%
```

### 决策类

```
结论 + 理由

例：选方案A 成本低速度快
例：不做 投入产出比差
例：推迟 资源不够
```

## 核心功能

### 1. 内容压缩

- 长文缩短句
- 去掉冗余信息
- 保留核心要点

### 2. 指令生成

- 动词开头
- 明确目标
- 限时要求

### 3. 状态汇报

- 结果优先
- 问题前置
- 数据说话

## 输入参数

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| content | String | 是 | 需要压缩的内容 |
| type | String | 否 | 输出类型：command/status/decision |

## 输出格式

```json
{
  "original": "我认为我们应该考虑一下这个方案的可行性",
  "brief": "评估方案可行性",
  "compressed": true,
  "wordCount": {
    "before": 16,
    "after": 6
  }
}
```

## 使用流程

```bash
# 压缩内容
brief "我认为我们应该考虑一下这个方案的可行性"
→ 评估方案可行性

# 指定类型
brief --type command "请修改一下这个接口，增加返回字段"
→ 改接口 加返回字段

# 状态汇报
brief --type status "测试已经全部通过了，覆盖率达到了85%"
→ 测试通过 覆盖率85%
```

## 最佳实践

1. **先给结论**：最重要的信息放最前面
2. **数字说话**：用数据代替形容词
3. **动词主导**：每个句子至少一个动词
4. **拒绝废话**：每句话都要产生价值
5. **保持简短**：单条信息不超过3句话
