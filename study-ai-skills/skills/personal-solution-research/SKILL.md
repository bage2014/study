# personal-solution-research

## 功能描述

方案调研技能，用于在方案设计、功能实现阶段进行多方案比对分析。通过收集业界标杆方案、大厂实现方案、开源方案等，进行系统性的对比分析，帮助团队做出最佳技术决策。每个调研至少包含3个方案，建议5个左右方案进行全面比对。

## 触发条件

- 需要设计新功能的技术方案时
- 需要调研某技术领域的实现方案时
- 需要在多个技术方案中做出选择时
- 需要评估方案的可行性和成本时

## 何时使用

- 项目立项阶段需要技术选型时
- 需求明确后需要设计技术方案时
- 遇到技术难题需要寻找解决方案时
- 需要对标业界标杆进行方案优化时
- 需要评估方案的投入产出比时

## 何时不使用

- 已有明确方案且经过充分论证时
- 简单功能实现无需多方案比对时
- 用户明确指定了技术方案时

## 核心功能

### 1. 方案收集

- 收集业界标杆方案（如 Google、Amazon、阿里、腾讯等大厂实现）
- 收集开源社区方案（GitHub、Apache、CNCF等）
- 收集行业最佳实践案例
- 收集团队内部历史方案

### 2. 方案比对分析

- 至少3个方案，建议5个方案进行比对
- 分析每个方案的优势和不足
- 评估方案的效益受益和成本
- 评估实现复杂度和维护运营成本

### 3. 方案推荐

- 给出推荐方案及理由
- 提供方案实施建议
- 提供风险提示和应对措施

### 4. 调研文档生成

- 生成结构化的调研文档
- 包含方案对比表格、分析结论、推荐建议
- 支持导出到 docs 目录

## 方案调研流程

```
明确调研目标 → 收集候选方案 → 建立评估维度 → 方案分析比对 → 生成调研报告 → 方案推荐
```

### 详细步骤

1. **明确调研目标**：确定调研的功能、性能、成本等要求
2. **收集候选方案**：收集至少3个，建议5个候选方案
3. **建立评估维度**：确定评估指标和权重
4. **方案分析比对**：逐个分析方案的优缺点
5. **生成调研报告**：输出结构化的调研文档
6. **方案推荐**：给出推荐方案和实施建议

## 评估维度

### 核心评估指标

| 维度 | 权重 | 说明 |
|------|------|------|
| **功能匹配度** | 25% | 方案是否满足需求功能 |
| **性能表现** | 20% | 响应时间、吞吐量、并发能力 |
| **可靠性** | 15% | 可用性、容错能力、数据一致性 |
| **成本** | 15% | 开发成本、运维成本、硬件成本 |
| **复杂度** | 10% | 实现难度、学习曲线、集成复杂度 |
| **生态支持** | 10% | 社区活跃度、文档完善度、第三方支持 |
| **扩展性** | 5% | 未来扩展能力、技术演进空间 |

### 详细评估项

#### 功能匹配度

- 核心功能覆盖程度
- 非核心功能支持情况
- 定制化能力
- 与现有系统集成能力

#### 性能表现

- 响应时间（P50/P95/P99）
- 吞吐量（TPS/QPS）
- 并发处理能力
- 资源利用率（CPU/内存/网络）
- 扩展性（水平扩展能力）

#### 可靠性

- 可用性（99.9%/99.99%/99.999%）
- 容错能力（故障恢复、降级策略）
- 数据一致性（ACID/最终一致性）
- 灾难恢复能力

#### 成本

- 开发成本（人力投入、时间周期）
- 运维成本（服务器、带宽、人力）
- 许可证成本（商业软件费用）
- 硬件成本（存储、计算资源）

#### 复杂度

- 实现难度（技术门槛、学习曲线）
- 集成复杂度（与现有系统对接）
- 部署复杂度（环境配置、依赖管理）
- 团队技术储备（现有技术栈匹配度）

#### 生态支持

- 社区活跃度（GitHub stars、issue响应速度）
- 文档完善度（官方文档、教程、案例）
- 第三方支持（插件、工具、服务）
- 商业支持（厂商服务、培训）

#### 扩展性

- 功能扩展能力（插件化、模块化）
- 性能扩展能力（水平扩展、垂直扩展）
- 技术演进空间（是否处于技术前沿）
- 多租户支持能力

## 使用流程

### 创建调研

```bash
# 创建方案调研
research create --topic "用户认证方案" --scope "调研OAuth2、JWT等方案" --requirement "支持多端登录"

# 指定方案数量
research create --topic "缓存方案" --schemeCount 5
```

### 查看调研

```bash
# 列出调研文档
research list

# 查看调研详情
research view docs/spec/认证方案调研-2026-06-25.md
```

### 更新调研

```bash
# 更新调研报告
research update docs/spec/认证方案调研-2026-06-25.md
```

## 最佳实践

1. **明确目标**：调研前明确调研目标和评估标准
2. **多方案收集**：至少收集3个方案，建议5个方案进行比对
3. **客观评估**：基于数据和事实进行评估，避免主观臆断
4. **权重分配**：根据实际情况调整评估维度的权重
5. **文档记录**：将调研结果整理成文档，便于团队共享和追溯

## 配置要求

无需额外配置，依赖 common-spec-driven 和 personal-doc-manager 技能。

## 扩展指南

1. 在评估维度表中添加新维度
2. 设置合理的权重
3. 更新方案对比表格和评分汇总

### 集成外部调研工具

可与以下工具集成：
- Google Trends：技术趋势分析
- GitHub Insights：开源项目活跃度分析
- Stack Overflow：技术问题热度分析

参考经验：方案调研应基于客观数据和事实，多维度评估，确保选择最适合当前场景的方案。
